# WeldSnap V2.0 系统日志与业务审计功能开发及踩坑经验总结

## 📖 一、 需求与架构背景

为了提升 WeldSnap 石化管道焊口质量录入系统的运维监控能力与关键业务追溯能力，我们为控制台开发了 **全链路系统日志与业务审计系统**。

整体设计分为三大板块：
1. **运维运行日志（Pino NDJSON）**：记录全量 HTTP 请求响应、SQL 初始化、环境变量及底层异常。
2. **自定义业务审计日志 (`AUDIT: 35`)**：引入专门的业务级别，使用易懂的自然语言记录“项目/管线/焊口增删改”、“照片直传/驳回/批量导出”、“用户权限变更”等纯业务事件。
3. **前端日志分析与控制组件 (`LogViewer.jsx`)**：提供 Carbon 系统风格的检索面板，支持按 TraceID、日志级别、关键字、时间范围筛选，支持全局动态日志级别控制与双格式导出。

## 🛠️ 二、 核心技术实现细节

### 1. Pino 自定义日志级别注册与语义助手
在 `src/lib/logger.js` 中配置 Pino 的 `customLevels`：
```javascript
const logger = pino({
  customLevels: { audit: 35 },
  useOnlyCustomLevels: false,
  // 级别映射: TRACE(10) < DEBUG(20) < INFO(30) < AUDIT(35) < WARN(40) < ERROR(50) < FATAL(60)
});
```

创建 `src/lib/audit.js` 统一辅助模块，结合 `AsyncLocalStorage` 自动注入上下文操作者姓名：
```javascript
const { logger } = require('./logger');
const { getTraceStore } = require('./trace');

function logAudit(action, description, details = {}) {
  const store = getTraceStore() || {};
  const operator = store.uploaded_by || '系统';
  const formattedMsg = `${operator} ${description}`;
  logger.audit({ type: 'audit', action, ...details }, formattedMsg);
}
```

### 2. 全量核心 API 业务日志埋点
在关键路由中统一注入语义化日志：
- `src/app/api/admin/projects/`：`系统管理员 新建了项目 "炼化一期" (施工号: SG-2026)`
- `src/app/api/admin/pipelines/`：`系统管理员 在项目 [炼化一期] 下新建了管线号 [PL-001]`
- `src/app/api/welds/`：`张师傅 在管线 [PL-001] 下现场创建了焊口号 [W-01]` / `批量删除了 15 条焊口记录`
- `src/app/api/upload/confirm/`：`张师傅 为管线 [PL-001] / 焊口 [W-01] 上传了 [组对] 工序照片`
- `src/app/api/admin/photo/reject/`：`系统管理员 将管线 [PL-001] / 焊口 [W-01] 的 [打底] 照片标记为不合格`
- `src/app/api/project/export-manifest/`：`系统管理员 请求打包下载了 10 道焊口的 30 张照片 ZIP 压缩包`
- `src/app/api/admin/users/`：`系统管理员 创建了新用户账号 "worker_02" (角色: 施工人员)`

### 3. 日志分析面板 (`LogViewer.jsx`)
- **勋章样式**：为 `AUDIT` 级别配置专属紫罗兰样式 (`bg: #f6f2ff`, `text: #8a3ff8`)。
- **快捷按钮**：新增 `[ 📑 仅看业务审计 ]` 和 `[ ⚠️ 仅看报错 ]` 一键切换。
- **绝对路径存储**：配合 `pino-roll` 按天轮转且限制单文件 10MB，自动清理 30 天前旧日志。

## 🚨 三、 踩坑经验与排查复盘 (Critical Lessons)

### 踩坑 1：`.gitignore` 模糊规则误杀源码路由 (致命陷阱)

- **现象**：本地开发和冒烟测试全部通过，但 `git push` 到 GitHub 触发 CI/CD 自动构建部署后，线上服务器访问系统日志控制台直接报 `GET /api/admin/logs 404 (Not Found)`。
- **排查过程**：
  检查 Next.js 代码完全正常，`pnpm build` 本地打包通过。但是检查仓库远端代码时，发现线上的 `server.js` 产物中根本不包含 `/api/admin/logs/` 路由！
  进一步检查 `git status --ignored`，发现 `src/app/api/admin/logs/` 居然出现在了 **Ignored Files** 列表里。
- **根因分析**：
  项目的 `.gitignore` 文件中配置了日志忽略规则：
  ```gitignore
  # 错误的配置
  logs/
  ```
  在 Git 的 Pathspec 语法中，**不带前置斜杠的 `logs/` 会递归匹配项目中任何层级下名为 `logs` 的目录**。
  因此，不仅项目根目录下的运行日志文件夹 `/logs/` 被忽略了，**连源码路径 `src/app/api/admin/logs/` 也被 Git 当作日志目录给自动忽略了**！这就导致 `git commit` 时 API 路由文件压根没进 Git 仓库，线上部署时自然报 404。
- **修复与避坑原则**：
  将 `.gitignore` 修正为严格的根目录限制：
  ```gitignore
  # 正确的配置 (带前置斜杠，仅忽略根目录 logs 文件夹)
  /logs/
  ```
  **教训**：在配置 `.gitignore` 忽略项目运行产生的日志（`logs/`）、数据（`data/`）、缓存（`cache/`）等目录时，**必须加上前置斜杠 `/`**（例如 `/logs/`），切勿写成相对匹配，防止误杀源码或组件路径。

### 踩坑 2：自定义日志级别校验白名单遗漏 (`VALID_LEVELS`)

- **现象**：在前端设置全局日志级别为 `AUDIT (业务审计专属)` 并点击应用时，页面弹出错误提示 `无效的日志级别: audit`。
- **根因分析**：
  在 `src/lib/logger.js` 中注册了 `customLevels: { audit: 35 }`，并在 Pino 模块的级别白名单中加入了 `audit`。但是，在后端日志级别控制接口 `src/app/api/admin/logs/level/route.js` 中，硬编码了请求校验白名单：
  ```javascript
  // 遗漏了 'audit'
  const VALID_LEVELS = ['trace', 'debug', 'info', 'warn', 'error', 'fatal'];
  ```
  当前端发起 POST 请求设置级别为 `audit` 时，校验直接拦截并返回了错误响应。
- **修复与避坑原则**：
  在 `VALID_LEVELS` 数组中补全 `'audit'`：
  ```javascript
  const VALID_LEVELS = ['trace', 'debug', 'info', 'audit', 'warn', 'error', 'fatal'];
  ```
  **教训**：系统中引入新的枚举值、常量或自定义扩展级别时，必须梳理并同步更新全链路中的所有校验白名单（Validation Schemas / Constants Arrays）。

---

### 踩坑 3：Standalone 打包模式下的日志路径解析与 CI/CD 目录保障

- **现象**：Next.js 在 standalone 模式打包发布后，如果是通过 PM2 或 systemd 在不同子目录下启动 `node server.js`，可能会导致 `pino-roll` 的相对路径 `'logs/weldsnap-run.log'` 发生读写偏移，导致前端 REST API 无法找到真正的日志文件。
- **修复方案**：
  1. 在 `src/lib/logger.js` 中使用 `process.cwd()` 显式绑定绝对路径：
     ```javascript
     const logFilePath = path.resolve(process.cwd(), 'logs', 'weldsnap-run.log');
     ```
  2. 在 GitHub Actions 部署脚本 `.github/workflows/deploy.yml` 的远程解压步骤中补全持久化目录创建指令：
     ```yaml
     - name: 8. SSH 远程解压与 PM2 启动
       script: |
         cd /var/www/WeldSnap
         tar -xzf release.tar.gz
         mkdir -p data
         mkdir -p exports
         mkdir -p logs  # 🚀 保证 CI 解压后日志目录必定就绪
     ```
- **避坑原则**：对于生产环境 Node.js 应用程序的文件读写，统一采用 `path.resolve(process.cwd(), ...)` 绑定运行根路径，并在 CI/CD 部署流程中显式创建持久化目录。

## 🎯 四、 总结与后续建议

1. **断言需严谨**：在向用户或团队提供架构保障承诺之前，必须全面检查 Git 状态、`.gitignore` 规则以及 CI/CD 构建脚本，避免因规则冲突导致“代码看似写完实则未入库”的尴尬。
2. **自动化测试覆盖**：保留并定期运行 `run_full_smoke_test.js` 脚本，模拟完整 API 交互并校验 `GET /api/admin/logs?auditOnly=true` 的审计日志产出，确保后续业务迭代不会破坏审计埋点。
